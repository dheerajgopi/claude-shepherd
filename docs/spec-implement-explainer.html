The spec-implement skill

A strict, hook-enforced workflow that turns a feature idea into committed, tested code — design first, requirements second, failing tests third, implementation last.

red → green  ·  nothing skips a step
The big idea

Three principles, enforced by the machine — not by good intentions

Most "AI writes a feature" flows jump straight to code. spec-implement refuses to. It forces a human-approved trail of what and why before a single line of production code is written — and uses mechanical guardrails so the rules can't be quietly broken.

📐

Spec-driven

A design sketch and EARS requirements are written and approved by a human before code exists. The spec is the source of truth — every test traces back to a requirement.

🔴🟢

Test-first (red → green)

Failing tests are written and committed red first. Implementation then runs until those exact tests pass — committed green. The test defines "done".

🔒

Hook-enforced

File-write boundaries are enforced by PreToolUse hooks, not prompts. The test-writing phase physically cannot touch production code, and vice-versa.

The pipeline

Four sequential loops

Each loop is a long-running agent session with one job. Click a stage to see what it does, what it produces, and where a human steps in.

How the human stays in control

A resumable checkpoint state machine

The engine runs headlessly. Whenever it needs a human decision, it exits with a specific code instead of guessing. The outer agent reads the code, asks you, and re-invokes — resuming the same session. The exit code is the protocol.

Why it's safe to trust

The guardrails

The rules aren't suggestions in a prompt — they're mechanical invariants the agent can't route around.

🔒 Path policy PreToolUse hook

A pure decision function classifies every file write. Loops 0–2 are allow-only (design/, requirements/, test files). Loop 3 is deny-under test paths — it can only write production code. A denial is a design decision, not an obstacle to retry around.

🧾 Automated commits exact format

Only the engine creates spec-implement(<slug>) commits: chore, red, red(n), green. They're audit artifacts of the choreography — the agent never hand-crafts one.

🧬 Traceability matrix coverage gate

Every test is tagged # requirement: <id>. Before the red commit, a verifier checks every requirement (and every example-table row) maps to a test. A gap stops the run for human judgment (exit 11).

🗂️ Machine-local workspace gitignored

The whole .shepherd/ workspace — config, design, requirements, state — is gitignored. Only the test and production code ever reach version control.

↩️ Escalation channel exit 12

If implementation reveals a test must change significantly, Loop 3 can't just edit it. It escalates for approval; an approved change re-enters Loop 1 + 2 and produces a fresh red(n) commit.

🔁 Resumable sessions prompt cache

Each re-invocation resumes the same SDK session (resume=<id>), preserving the cache prefix and the agent's context across every human checkpoint.
